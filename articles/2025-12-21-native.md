## はじめに

エンジニア歴9年、Swift歴7年、フリーランス歴5年の私が、Swift → Flutter → React Native とモバイル開発技術を渡り歩いてきた経験をもとに、**なぜ React Native にかけてみることにしたのか**をお話しします。

各技術の使用期間は以下の通りです。

- **Swift**: 2018年〜2024年（約7年）
- **Flutter**: 2025年前半
- **React Native**: 2025年後半〜

それぞれの技術に良さがあり、どれも素晴らしい技術です。しかし、**私が個人的に予想する今後の開発トレンド（少数エンジニア × Web・バックエンド・ネイティブアプリのモノレポ × 生成AI）** を考えると、React Native にかけてみる価値があると感じました。

---

## 私の技術遍歴

### Swift 時代（2018年〜2024年）

私のモバイル開発キャリアは Swift から始まりました。約7年間、Swift をメイン技術として業務に携わってきました。

Swift は言語として非常に好みで、シンプルな開発体験としては最高峰だと感じていました。

### Flutter 時代（2025年前半）

2024年末、フリーランス市場での Swift 案件が減少していたタイミングで、Flutter のリードエンジニアの誘いがありました。元々一度 Flutter を経験してみたかったこともあり、案件に参画しました。

正直、Dart という言語に慣れてしまえば、Flutter に対してネガティブな印象はほとんどなく、やりやすかったです。

### なぜ Swift に戻らなかったか

Flutter の案件が一区切りついた2025年6月頃、一度 Swift に戻ろうと考えました。それまでフリーランスとして活動する中で、巨大企業である Apple が牽引してくれることを信頼しており、Swift に乗っかっていれば間違いないだろうと思っていました。

しかし、その頃には生成AIによるコード開発が本格化しており、状況は一変していました。

この頃流行っていた生成AI系の SDK は、TypeScript や Python での公開が中心で、Swift Package Manager での公開は後回しにされることが多く、Swift を続けていると置いていかれるような感覚がありました。

また、Xcode での生成AI開発体験も課題でした。GitHub Copilot for Xcode も存在しますが、Cursor や VSCode のエージェントプラグインと比べると、遅れをとっている印象でした。

さらに、フリーランス市場での Swift 案件も減少傾向にあり、Swift があと数年後も同じように使われ続けるか怪しいと感じました。

Apple が生成AI分野で明らかに遅れを取っている現状を見て、このまま Swift に依存し続けることはリスクだと判断しました。そこで Swift には戻らず、Flutter か React Native のどちらかを選択することにしました。

## 今後の開発トレンド予測

Flutter か React Native か選ぶ上で、今後の開発トレンドについて考えました。

### 少数エンジニアでモノレポ開発する時代

- 開発エンジニアの数は減少し、一人でカバーするフィールドが広くなる
- コーディングエージェントを活用した開発が主流に
- **Web・バックエンド・ネイティブアプリを一つのモノレポで管理**することが、トークン効率的にも開発効率的にも最適
- 仕様書もコードも全部一箇所で管理し、少数精鋭で開発

### TypeScript 統一がスタンダードに

このような時代に選択される言語は、**TypeScript 統一**がスタンダードになっていくのではないかと考えています。

- バックエンドも TypeScript（Node.js/Bun）
- Web も TypeScript（React）
- ネイティブアプリも TypeScript（React Native）

生成AI時代において、TypeScript を触り続けることは合理的な選択です。また、TypeScript ができればフロントエンドもバックエンドもネイティブアプリも書けるため、採用面でも有利になると考えています。

### クロスプラットフォームの必然性

「LLM が進化したら、どの言語でも同じだろう」という意見もあるかもしれません。しかし、仮にそうなったとしても：

- わざわざ Swift と Kotlin で2つのコードを生成する必要があるか？ → 出力トークン量が2倍
- 一人のエンジニアが Swift と Kotlin を両方管理するなら、同じようなアーキテクチャや実装パターンに揃えた方が認知コストは低い
- しかし揃えようとすると、Swift らしい書き方ができなくなる（逆もしかりで、Kotlin らしい書き方もできなくなる）
- 結局、2つの異なる形式のコードをメンテナンスする認知コスト・修正コストは高い

結局、iOS と Android の両方を作るなら、**最初からクロスプラットフォームを採用するのが良い**のではないでしょうか。そして、クロスプラットフォームの中でも、サーバーサイドとの言語統一や Web 開発のエコシステムに乗れるという点で、React Native に分があると考えています。

### 生成AI開発環境の優位性

VSCode 周りの生成AI エージェント拡張機能は非常に充実しています（Cursor、GitHub Copilot、Cline など）。また、Web 系の MCP サーバーも豊富で、Google Devtools MCP や Playwright MCP などを活用した開発が可能です。

React Native は Web としてもデプロイできるため、これらの恩恵を直接受けられます。一方、Xcode はこの点で遅れを取っており、生成AI を活用した開発体験には差があります。

## 各技術を触った感想

### Swift

**良い点**:
- 言語として非常に好み（Associated Value を持てる Enum、強力な型システムなど）
- SwiftUI + Combine の双方向データバインディングは本当に洗練されている
- シンプルな開発体験としては最高
- ビルトインコンポーネントが充実しており、別途ライブラリを探す必要がない

**課題**:
- iOS/Android 別開発のコスト問題
- 生成AI時代の開発体験（Xcode）
- 日本でのフリーランス市場での案件が減少傾向

### Flutter

**良い点**:
- クロスプラットフォームで iOS/Android 両対応できる
- 初心者でも取り組みやすい（ドキュメントが手厚く、使うライブラリも限られていて迷わない）
- Riverpod による状態管理が強力

**課題**:
- Dart エコシステムの未発達（TypeScript/JavaScript と比べると）
- 世界的なトレンドでは React Native が追い風の状況

### React Native

**良い点**:
- クロスプラットフォームで iOS/Android 両対応できる
- Next.js と Expo でコードを共通化でき、Web アプリとネイティブアプリを同時に開発できる
- Web としてもデプロイ可能なので、Storybook や Playwright での E2E テスト、ブラウザ操作系の MCP サーバーを使った開発など、Web 開発のエコシステムにそのまま乗れる
- TanStack Query、Vitest など Web 開発で培われたライブラリがそのまま使える
- パフォーマンスも向上しており、正直 Swift/Flutter と比べてそこまで差を感じない
- Expo Modules を使えば、いざというときに Swift/Kotlin のネイティブコードを組み込める

**課題**:
- Swift や Flutter に比べてライブラリのアップデートが大変で、メンテナンスコストがかかる（Expo の依存関係が複雑）
- まだ成熟しきっていない分野も多く、依存ライブラリも多いため、何かとバグを踏みやすい
- Swift/Kotlin/Flutter エンジニアだと Node.js/TypeScript 周りでつまずくポイントが多い（npm/pnpm/yarn/bun、package.json、tsconfig など）
- ライブラリの選択肢が膨大すぎる（メリットでもあるが、選定に迷う）

結局、UI を作るときに重要なのは「API の呼び出しと状態管理をどうするか」です。Swift だろうが Flutter だろうが React Native だろうが、そこは同じです。

### 比較表

| 観点 | Swift | Flutter | React Native |
|------|:-----:|:-------:|:------------:|
| クロスプラットフォーム対応 | ✕ | ◎ | ◎ |
| サーバーサイドとの言語統一 | △ | △ | ◎ |
| 生成AI時代のDX | △ | ○ | ◎ |
| Web 版の同時開発 | ✕ | △ | ◎ |
| ライブラリの豊富さ | ○ | △ | ◎ |
| 学習コスト（Web経験者） | △ | ○ | ◎ |
| 学習コスト（初心者） | ○ | ◎ | △ |
| メンテナンスコスト | ◎ | ○ | △ |
| パフォーマンス | ◎ | ◎ | ○ |
| 言語の好み | ◎ | ○ | ○ |

### それぞれを選ぶべきケース

それぞれの技術を選ぶべきケースをまとめます。

- **Swift/Kotlin**: iOS または Android 単体のアプリをすぐ作りたい場合は最速。メンテナンスコストが少なく安定しており、パフォーマンスも最高。ネイティブ SDK が必須な場合も選択肢になる。既存資産がありロジック部分を共通化したい場合は KMP も検討
- **Flutter**: 初心者や非モバイルエンジニアが多いチームでクロスプラットフォームを始めたい場合。学習コストが低くドキュメントも手厚い
- **React Native（Expo）**: Web 版も同時に開発したい場合、サーバーサイドと言語を統一したい場合、生成AI時代の開発体験を重視する場合

## なぜ React Native にかけてみることにしたのか

上記のトレンド予測と各技術の比較を踏まえ、Flutter か React Native かという選択において、私は **「今後は TypeScript の時代になるのでは」** という考えから React Native を選びました。

理由をまとめると：

1. **少数エンジニアでモノレポ開発する時代に向いていそう**
2. **TypeScript 統一で Web・バックエンド・ネイティブアプリを一貫して開発できる**
3. **Web を取り巻くエコシステムの成長が早く、そこにただ乗りできる**
4. **生成AI開発（Vibe Coding）との親和性が高い**（VSCode + MCP エコシステムの恩恵、Vibe Coding ツールの多くが Expo ベース）
5. **世界的なトレンドで React Native が追い風**（日本では Flutter が優勢だが、React vs Vue と同様に世界の動向には抗えない）
6. **Apple/Swift だけに技術的に依存するリスクを回避できる**

React Native の辛いところ（Expo や React Native のライブラリのアップデートの大変さ、学習コストの高さ）は確かにあります。しかし、それを差し引いても、今後の開発トレンドを考えると React Native にかけてみる価値があると感じています。

そこで React Native の案件を探し、現在はプロジェクトに参加しています。

TypeScript 時代に乗る覚悟で、これからも React Native で開発を続けていきます。
