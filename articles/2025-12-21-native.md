## はじめに

エンジニア歴9年、フリーランス歴6年の私が、Swift → Flutter → React Native とモバイル開発技術を渡り歩いてきた経験をもとに、**今後ネイティブアプリを開発するなら React Native を選ぶ**という結論に至った理由をお話しします。

各技術の使用期間は以下の通りです。

- **Swift**: 2018年〜2024年（約7年）
- **Flutter**: 2025年前半
- **React Native**: 2025年後半〜

それぞれの技術に良さがあり、どれも素晴らしい技術です。しかし、**私が個人的に予想する今後の開発トレンド（少数エンジニア × モノレポ × 生成AI）** を考えると、React Native を選択するのが最も合理的だと考えています。

---

## 私の技術遍歴

### Swift 時代（2018年〜2024年）

私のモバイル開発キャリアは Swift から始まりました。約7年間、Swift をメイン技術として業務に携わってきました。

Swift は言語として非常に好みです。Associated Value を持てる Enum や強力な型システムなど、書いていて気持ちが良い。特に SwiftUI と Combine を組み合わせた双方向データバインディングは、非常に洗練されていて書き心地が良いです。ネイティブアプリ開発用に設計された言語だけあって、ビルトインのUIコンポーネントも充実しており、シンプルな開発体験としては最高峰だと感じていました。

### Flutter 時代（2025年前半）

2024年末、フリーランス市場で魅力的な0→1のSwift案件が減少していたタイミングで、友人から Flutter のリードエンジニアの誘いがありました。元々一度 Flutter を経験してみたかったこともあり、案件に参画しました。

Flutter は非常に良い技術です。

- **初心者が最も挫折しにくい**: ドキュメントが手厚く、わかりやすい
- **使うライブラリがほぼ決まっている**: 選択肢で迷うことが少ない
- **Riverpod による状態管理が強力**: 非常によくできている
- **Flutter Hooks**: 後に React を学ぶ際に役立った

正直、Dart という言語に慣れてしまえば、Flutter に対してネガティブな印象はほとんどなく、やりやすかったです。

### なぜ Swift に戻らなかったか

Flutter の案件が一区切りついた2025年6月頃、一度 Swift に戻ろうと考えました。それまでフリーランスとして活動する中で、巨大企業である Apple が牽引してくれることを信頼しており、Swift に乗っかっていれば間違いないだろうと思っていました。

しかし、その頃には生成AIによるコード開発が本格化しており、状況は一変していました。

問題は、**Xcode での生成AI開発体験が非常に悪い**ということです。GitHub Copilot for Xcode も存在しますが、Cursor や VSCode のエージェントプラグインと比べると、はるかに使いにくい。

さらに、生成AIを活用した SDK は基本的に TypeScript で先に公開される状況でした。最近のSDKはまず TypeScript で作られ、他言語は後回しになることがほとんどです。Swift で使える生成AI系 SDK の選択肢としては、Apple 純正の Apple Intelligence SDK がありますが、正直なところ Claude（Anthropic）や OpenAI、Google に大きく遅れを取っており、魅力的とは言えません。

Apple が生成AI分野で明らかに遅れを取っている現状を見て、このまま Swift に依存し続けることはリスクだと判断しました。そこで Swift には戻らず、Flutter か React Native のどちらかを選択することにしました。

### React Native 時代（2025年後半〜）

Flutter か React Native か。私は React Native を選びました。理由は明確です。

**「今後は TypeScript の時代だ」**

いくつかの要因がこの判断を後押ししました。

1. **Vibe Coding ツールの多くが React Native（Expo）ベース**: 生成AIでネイティブアプリのコードを生成するツールのほとんどが、React Native + Expo をベースにしていた
2. **世界的なトレンド**: 日本では Flutter が優勢だが、世界的には React Native が採用されている。React vs Vue の戦いで、最終的に React が勝ったように、世界の動向には抗えない
3. **SDK の公開順序**: 各種 SDK は基本的に TypeScript や Python で先に公開され、後から他言語に展開される

## 各技術を触った感想

### Swift

**良い点**:
- 言語として非常に好み（Associated Value を持てる Enum、強力な型システムなど）
- SwiftUI + Combine の双方向データバインディングは本当に洗練されている
- シンプルな開発体験としては最高
- ネイティブアプリ用に作られているため、ビルトインコンポーネントが充実
- コンポーネントライブラリを別途探す必要がない

**課題**:
- iOS/Android 別開発のコスト問題
- 生成AI時代の開発体験（Xcode）

### Flutter

**良い点**:
- クロスプラットフォームで iOS/Android 両対応できる
- 初心者に最もおすすめ
- ドキュメントが手厚くわかりやすい
- 使うライブラリがほぼ限られていて迷わない
- Riverpod が非常に強力
- とにかくやりやすい

**課題**:
- Dart エコシステムの未発達（TypeScript/JavaScript と比べると）
- 世界的なトレンドでは React Native が追い風の状況

### React Native

**良い点**:
- クロスプラットフォームで iOS/Android 両対応できる
- Web エコシステムが使える（Storybook、Playwright、Vitest、TanStack Query など）
- パフォーマンスも向上しており、正直 Swift/Flutter と比べてそこまで差を感じない

**課題**:
- Node.js/TypeScript を知らない人がつまずくポイントが多い（npm/pnpm/yarn/bun、package.json、tsconfig など）
- ライブラリの選択肢が膨大すぎる
- React を学ぶ必要がある
- SDK のバージョンアップが辛い（Expo の依存関係が複雑）

正直、Swift エンジニアにとっては、この学習曲線は非常に厳しいです。Web エンジニアが当たり前のように扱っているツールチェーン（なんちゃら.js 系のツール群）が、次から次へと出てきます。

結局、UI を作るときに重要なのは「API の呼び出しと状態管理をどうするか」です。Swift だろうが Flutter だろうが React Native だろうが、そこは同じ。そして React Native には TanStack Query があり、Storybook もあり、Web 開発で培われた膨大なエコシステムが使えます。

## 今後の開発トレンド予測

ここからは私の予測ですが、今後の開発は以下のようになると考えています。

### 少数エンジニアでモノレポ開発する時代

- 開発エンジニアの数は減少し、一人でカバーするフィールドが広くなる
- コーディングエージェントを活用した開発が主流に
- **Web・バックエンド・ネイティブアプリを一つのモノレポで管理**することが、トークン効率的にも開発効率的にも最適
- 仕様書もコードも全部一箇所で管理し、少数精鋭で開発

### TypeScript 統一がスタンダードに

このような時代に選択される言語は、**TypeScript 統一**がスタンダードになるはずです。

- バックエンドも TypeScript（Node.js/Bun）
- Web も TypeScript（React）
- ネイティブアプリも TypeScript（React Native）

各種 SDK も基本的に TypeScript か Python で先に公開されます。生成AI時代において、TypeScript を触り続けることは合理的な選択です。

### クロスプラットフォームの必然性

「LLM が進化したら、どの言語でも同じだろう」という意見もあるかもしれません。しかし、仮にそうなったとしても：

- わざわざ Swift と Kotlin で2つのコードを生成する必要があるか？ → 出力トークン量が2倍
- 一人のエンジニアが Swift と Kotlin を両方管理するなら、同じようなアーキテクチャや実装パターンに揃えた方が認知コストは低い
- しかし揃えようとすると、Swift らしい書き方ができなくなる（逆もしかりで、Kotlin らしい書き方もできなくなる）
- 結局、2つの異なる形式のコードをメンテナンスする認知コスト・修正コストは高い

結局、iOS と Android の両方を作るなら、**最初からクロスプラットフォームを採用するのが吉**です。

## 結論：なぜ React Native なのか

改めて、私が React Native を選んだ理由をまとめます。

1. **少数エンジニアでモノレポ開発する時代に最適**
2. **TypeScript 統一で Web・バックエンド・ネイティブアプリを一貫して開発**
3. **Web を取り巻くエコシステムの成長が早く、そこにただ乗りできる**
4. **生成AI開発（Vibe Coding）との親和性が高い**

React Native の辛いところ（SDK バージョンアップの大変さ、学習コストの高さ）は確かにあります。しかし、それを差し引いても、今後の開発トレンドを考えると React Native が最も合理的な選択だと考えています。

## 補足：それぞれを選ぶべきケース

とはいえ、すべてのケースで React Native が正解というわけではありません。

- **Swift**: iOS だけのアプリをすぐ作りたい場合は最速。Swift が一番良い開発体験
- **Flutter**: 初心者チーム、未経験者中心のプロジェクト。ドキュメントの手厚さと学習曲線の緩やかさは大きなメリット
- **Kotlin**: Android タブレット専用アプリなど、Android のみをターゲットにする場合
- **KMP（Kotlin Multiplatform）**: 既存の Swift/Kotlin 資産があり、内部コンポーネントを共通化したい場合のみ。それ以外なら Flutter か React Native の方がベター

---

## まとめ

各技術にはそれぞれの良さがあります。

- Swift は言語として好みで、開発体験もシンプル
- Flutter は初心者に優しく、ドキュメントも充実
- React Native は学習コストは高いが、Web エコシステムの恩恵を受けられる

しかし、**私が個人的に予想する今後の開発トレンド（少数エンジニア × モノレポ × 生成AI）** を考えると、私は React Native を選択しました。

TypeScript 時代に乗る覚悟で、これからも React Native で開発を続けていきます。
