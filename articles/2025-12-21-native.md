## TL;DR

- Swift/Flutter/React Native を経験した結果、少人数 × モノレポ × 生成AIの流れを踏まえると React Native にかけてみる価値はあると感じています
- TypeScript 統一と Web エコシステム、生成AIツールの恩恵が大きい
- 依存関係の更新や学習コストの課題はあるため、検証しながら進める前提

## はじめに

エンジニア歴9年、Swift歴7年、フリーランス歴5年の私が、Swift → Flutter → React Native とモバイル開発技術を渡り歩いてきた経験をもとに、**なぜ React Native にかけてみることにしたのか**をお話しします。

各技術の使用期間は以下の通りです。

- **Swift**: 2018年〜2024年（約7年）
- **Flutter**: 2025年前半
- **React Native**: 2025年後半〜

それぞれの技術に良さがあり、どれも魅力のある技術です。以下では、**私が個人的に予想する今後の開発トレンド（少数エンジニア × Web・バックエンド・ネイティブアプリのモノレポ × 生成AI）** と、その上での判断理由を整理します。

## 私の技術遍歴

### Swift 時代（2018年〜2024年）

私のモバイル開発キャリアは Swift から始まりました。約7年間、Swift をメイン技術として業務に携わってきました。

Swift は言語として非常に好みで、シンプルな開発体験としてはかなり高い水準だと感じていました。

### Flutter 時代（2025年前半）

2024年末、フリーランス市場での Swift 案件が減少していたタイミングで、Flutter のリードエンジニアから誘いがありました。元々一度 Flutter を経験してみたかったこともあり、案件に参画しました。

正直、Dart という言語に慣れてしまえば、Flutter に対してネガティブに感じることはほとんどなく、全体的にやりやすかったと感じました。

### なぜ Swift に戻らなかったか

Flutter の案件が一区切りついた2025年6月頃、一度 Swift に戻ろうと考えました。それまでフリーランスとして活動する中で、巨大企業である Apple が牽引してくれることを信頼しており、Swift に乗っかっていれば大きくは外さないだろうと思っていました。

しかし、その頃には生成AIによるコード開発が本格化しており、状況は大きく変わっていると感じました。

この頃流行っていた生成AI系の SDK は、TypeScript や Python での公開が中心で、Swift Package Manager での公開は後回しにされることが多く、Swift を続けていると取り残されるかもしれないという不安がありました。

さらに、フリーランス市場での Swift 案件も減少傾向にあり、Swift があと数年後も同じように使われ続けるか不透明だと感じました。

Apple が生成AI分野で出遅れていると感じたため、このまま Swift に依存し続けることはリスクだと判断しました。そこで Swift / Flutter / React Native のどれを軸にしていくかを検討しました。
この経験を踏まえ、個人的に今後の流れを整理すると次のようになります。

## 今後の開発トレンド予測

Swift / Flutter / React Native のどれを軸にしていくか選ぶ上で、今後の開発トレンドについて考えました。

### 少数エンジニアでモノレポ開発する時代

- 開発エンジニアの数は減少傾向になり、一人でカバーするフィールドが広くなる
- コーディングエージェントを活用した開発が増えていく
- **Web・バックエンド・ネイティブアプリを一つのモノレポで管理**することが、トークン効率の観点でも開発効率的にも相性が良い

### TypeScript 統一がスタンダードに

このような時代に選択される言語は、**TypeScript 統一**が有力な選択肢の一つになっていくのではないかと考えています。

- バックエンドも TypeScript（Node.js/Bun）
- Web も TypeScript（React）
- ネイティブアプリも TypeScript（React Native）

言語が揃うと開発・レビュー・採用の文脈が揃いやすく、運用も楽になりやすいので、生成AI時代に TypeScript を触り続けることは合理的だと感じています。フロントエンド/バックエンド/ネイティブを同じ言語で書きやすく、採用面でもプラスに働きやすいと考えています。

### Swift/Kotlin よりもクロスプラットフォームを選ぶ理由

まず結論として、iOS と Android の両方を作る前提なら、**最初からクロスプラットフォームを選ぶ価値は高まりやすい**と感じています。

「LLM が進化したら、どの言語でも同じだろう」という意見もあるかもしれません。しかし、仮にそうなったとしても、次の点は残ると考えています：

- わざわざ Swift と Kotlin で2つのコードを生成する必要があるか？ → 出力トークン量が倍近く
- 一人のエンジニアが Swift と Kotlin を両方管理するなら、同じようなアーキテクチャや実装パターンに揃えた方が認知コストは低い
- しかし揃えようとすると、Swift らしい書き方ができなくなる（逆もしかりで、Kotlin らしい書き方もできなくなる）
- 結局、2つの異なる形式のコードをメンテナンスする認知コスト・修正コストは高くなりやすい

### 新しい言語を習得するコストは近い

ここで言うコストは、言語仕様だけでなく、ツールやエコシステムの学習も含めた感覚です。

そもそも iOS/Android の開発は、同じ人がやる方が効率的に回ることが多いです。今までは Swift と Kotlin という言語の壁があり、それぞれ専門のエンジニアが担当するのが一般的でした。

しかし、生成AIによる開発効率の向上に伴い、一人のエンジニアがカバーする範囲は広がっています。生成AIがある今では、Swift エンジニアが Kotlin も見る（逆もしかり）というのは現実的な選択肢になりつつあります。

つまり、Swift エンジニアとして今後も活躍するためには、Kotlin も習得する必要性が出てくるということです。どうせ新しい言語を習得するなら、React Native（TypeScript）を選ぶコストも許容範囲内ではないでしょうか。

もちろん、React Native は Web 技術ベースなので、Web の React エンジニアがモバイル開発に転身しやすく、そことの競争にはなります。しかし、ネイティブモジュールの組み込みやストアリリースの経験など、ネイティブ開発をやってきた人には一定の優位性があります。極端に悲観する必要はないと考えています。

### 生成AI開発環境の差

VSCode 周りの生成AI拡張やWeb系MCP（Playwright など）が充実しており、Web 技術ベースの開発は恩恵を受けやすいと感じています。Xcode では同等の体験を得にくく、ここも判断材料でした。

## なぜ React Native にかけてみることにしたのか

上記のトレンド予測を踏まえ、私が React Native を選んだ理由は以下の通りです：

1. **少数エンジニアでモノレポ開発する時代に向いていそう**
2. **TypeScript 統一で Web・バックエンド・ネイティブアプリを一貫して開発できる**
3. **Web を取り巻くエコシステムの成長が早く、その恩恵を受けやすい**
4. **生成AI開発（Vibe Coding）との親和性が高い**（VSCode + MCP エコシステムの恩恵、Vibe Coding ツールの多くが Expo ベース）
5. **世界的に見ると React Native に勢いがある**（日本では Flutter が優勢だが、React vs Vue と同様に世界の潮流の影響は大きい）
6. **Apple/Swift だけに技術的に依存するリスクを下げられる**

React Native にも課題はありますが、今後の開発トレンドを考えると挑戦する価値はあると感じており、今は React Native のプロジェクトに参加しています。

## 補足：各技術を触った感想

### Swift

**良い点**:
- 言語として非常に好み（Associated Value を持てる Enum、強力な型システムなど）
- SwiftUI + Combine の双方向データバインディングはとても洗練されている
- シンプルな開発体験としてはかなり高い水準
- ビルトインコンポーネントが充実しており、別途ライブラリを探す必要がない

**課題**:
- iOS/Android 別開発のコスト問題
- 生成AI時代の開発体験（Xcode）
- 日本でのフリーランス市場での案件が減少傾向

### Flutter

**良い点**:
- クロスプラットフォームで iOS/Android 両対応できる
- 初心者でも取り組みやすい（ドキュメントが手厚く、使うライブラリも限られていて迷わない）
- Riverpod による状態管理が強力だと感じています

**課題**:
- Dart エコシステムの未発達（TypeScript/JavaScript と比べると）
- Web 版の同時開発が難しい（技術的には可能だが、実用レベルではまだ課題がある）

### React Native

**良い点**:
- クロスプラットフォームで iOS/Android 両対応できる
- Next.js と Expo でコードを共通化でき、Web アプリとネイティブアプリを同時に開発できる
- Web としてもデプロイ可能なので、Storybook や Playwright での E2E テスト、ブラウザ操作系の MCP サーバーを使った開発など、Web 開発のエコシステムにそのまま乗れる
- TanStack Query、Vitest など Web 開発で培われたライブラリがそのまま使える
- パフォーマンスも向上しており、感覚としては Swift/Flutter と比べて大きな差は感じていません
- Expo Modules を使えば、いざというときに Swift/Kotlin のネイティブコードを組み込める

**課題**:
- Swift や Flutter に比べてライブラリのアップデートが大変で、メンテナンスコストが上がりやすい（Expo の依存関係が複雑）
- まだ成熟しきっていない分野も多く、依存ライブラリも多いため、何かとバグを踏みやすい
- Swift/Kotlin/Flutter エンジニアだと Node.js/TypeScript 周りでつまずくポイントが多い（npm/pnpm/yarn/bun、package.json、tsconfig など）
- ライブラリの選択肢が膨大で（メリットでもあるが、選定に迷う）

### それぞれを選ぶべきケース

それぞれの技術を選ぶべきケースをまとめます。

- **Swift/Kotlin**: iOS または Android 単体のアプリをすぐ作りたい場合は最短ルート。メンテナンスコストが少なく安定しており、パフォーマンスも高い。ネイティブ SDK が必須な場合も選択肢になる。既存資産がありロジック部分を共通化したい場合は KMP も検討
- **Flutter**: 初心者や非モバイルエンジニアが多いチームでクロスプラットフォームを始めたい場合。学習コストが低くドキュメントも手厚い
- **React Native（Expo）**: Web 版も同時に開発したい場合、サーバーサイドと言語を統一したい場合、生成AI時代の開発体験を重視する場合
